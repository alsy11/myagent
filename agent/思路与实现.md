## 前期调研

根据老师给我的开源文件，我大致理解如下

**[claw-code](https://github.com/ultraworkers/claw-code)** — 这是 Claude Code 的源码，项目结构非常清晰：`runtime/`（核心循环）、`tools/`（工具注册和执行）、`api/`（LLM 调用）、`commands/`（斜杠命令）。Claude Code 本质上是个 while 循环套着工具调用。

**[how-claude-code-works](https://github.com/Windy3f3f3f3f/how-claude-code-works)** — 这是理解claude code机制原理的一份文档，里面有几个观点我觉得特别有价值，是我第一次了解：

- Claude Code 有 7 种不同的「继续策略」来处理各种故障，用户根本看不到大多数错误，因为系统内部已经消化了
- 上下文压缩分 4 个级别：裁剪 → 折叠 → 摘要 → 子 Agent 接管，一步步来，不是暴力清空
- 安全检查用的是语法树分析，不是正则，有 23 项检查

**[anthropic/claude-code](https://github.com/anthropics/claude-code)** — 官方仓库，任务分三个阶段——收集上下文、执行操作、验证结果，这三个阶段是反复交替的。

﻿## 任务与 Claude Code 的区别

Claude Code 需要具备和用户持续交互的能力，需要处理自然语言或结构化输入，并且通常要经过多轮决策才能把任务处理好。它至少包含七个必要组件，流程大致如下：

```
用户说一句话
  ↓
模型想一想，决定要不要用工具
  ↓
如果要用工具 → 执行工具 → 把结果告诉模型
  ↓
模型再想一想……
  ↓
直到模型说"我做完了"，循环结束
```

而我的任务目标相对更聚焦：主要处理 Markdown / JSON 输入，任务边界更明确，不需要复杂的人机多轮对话，核心流程如下：

```
用户输入（Markdown / JSON）
   ->
解析输入 -> ArchitectureModel
   ->
工具调用（写文件、生成目录、生成路由、生成服务）
   ->
输出完整项目
```

## 项目实现思路

学习 Claude Code 的设计以及思路，然后用 DeepSeek API 搭一个能读架构文档、自动生成整个项目的 Agent。

### 输入

1. `Architecture_Documentation.md`

- 内容：系统说明、组件职责、技术栈、SQL、OpenAPI、测试用例、需求追踪矩阵等。
- 作用：作为 Agent 的主要信息来源，用于提取结构化架构模型。

2. `Architecture_View.md`

- 内容：UML / PlantUML 架构视图。
- 作用：提供系统图信息，转为可落地的 PlantUML 代码块。

Agent 启动的时候，先把这两个文件的内容读进来，放入系统提示词（system prompt）里

### 工具：4 个函数

Agent 给模型提供了 4 个工具：

| 工具            | 干什么                                   |
| --------------- | ---------------------------------------- |
| `write_file`    | 把内容写到 `generated/` 目录下的某个文件 |
| `read_file`     | 读取 `docs/` 目录下的架构文档            |
| `list_files`    | 列出已经生成了哪些文件                   |
| `task_complete` | 告诉 Agent 我做完了，循环结束            |

模型在生成项目的过程中，主要就是一直调 `write_file`，一个文件一个文件地写

### 系统提示词：告诉模型要生成什么

系统提示词里写清楚了两件事：

1. 要生成哪 21 个文件（按顺序），以及每个文件的关键要求
2. 几条硬规定：比如 `public/index.html` 必须是一个完整的、能独立运行的游戏页面，不能用省略号代替内容，不能截断

## 收获

 Claude Code 的核心 runtime 是一个 append-only 的消息列表。 Agent 没有单独的「状态管理」，所有状态都在 `messages` 数组里。每次调用 API，就把整个对话历史带上。这样做的好处是：调试方便（历史完整可查），出错了也可以从某个点重试。

如果某个工具调用失败，Agent 不会直接退出，而是把错误信息作为 `tool_result` 返回给模型，让模型自己决定怎么处理。Claude Code 的 Agent 循环有 7 种「继续策略」，大多数错误用户根本看不到，因为系统内部已经处理掉了。

从 Claude Code 的 prompt 设计里学到系统提示词要非常具体，Claude Code的系统提示词非常长、非常细，覆盖了大量边界情况。我的项目里，告诉模型要生成 21 个文件，必须用 `write_file` 一个一个写，不能用省略号，`public/index.html` 必须是完整的 HTML+CSS+JS

## 项目后续展望

目前我的项目是“一次性生成”模式：输入后生成完整项目并结束。相较 Claude Code，用户不能在生成过程中持续介入和修正。只是实现了核心思路中的while 循环 + 工具调用 + 消息历史即状态

后续开源实现：

- **上下文压缩**：裁剪 → 折叠 → 摘要 → 子 Agent 接管。当对话太长超出 context window 时，Claude Code 不是直接报错，而是分级压缩，用户完全无感知
- **权限系统**：Claude Code 有 5 层权限防御，包括语法树级别的 Shell 命令分析（23 项安全检查），不是简单的正则匹配
- **多 Agent 协调**： 子 Agent 并行处理子任务，但同时最多只允许 1 个子 Agent，防止失控
- **流式输出**：Claude Code 是实时流式渲染的，字符一个一个出现；我的是等整个响应回来再处理
- **会话持久化**：Claude Code 支持 `--resume` 恢复上次的会话，我的每次都是全新开始
